# Metodologías de desarrollo: Agile, Scrum y Kanban

## ¿Qué es Agile?

Agile es una filosofía o conjunto de principios para el desarrollo de software que busca adaptarse al cambio y entregar valor de forma continua. A diferencia de los modelos tradicionales (como el modelo en cascada), Agile no se basa en planificar todo desde el inicio, sino en trabajar de forma iterativa y flexible.

El objetivo principal de Agile es desarrollar software útil de manera progresiva, permitiendo incorporar cambios incluso en fases avanzadas del proyecto.

Los principios fundamentales de Agile incluyen:

* Entrega frecuente de software funcional
* Adaptación constante a los cambios
* Comunicación continua con el cliente
* Colaboración entre los miembros del equipo
* Priorizar el software funcionando sobre la documentación excesiva

En resumen, Agile permite reducir riesgos, mejorar la calidad del producto y responder mejor a las necesidades reales del usuario.

---

## ¿Qué es Scrum?

Scrum es un marco de trabajo dentro de Agile que organiza el desarrollo en ciclos cortos llamados *sprints*, normalmente de 1 a 4 semanas.

### Roles principales en Scrum

* **Product Owner**: define qué se debe desarrollar y prioriza las tareas (backlog)
* **Scrum Master**: facilita el proceso y elimina impedimentos
* **Equipo de desarrollo**: construye el producto

### Conceptos clave

* **Sprint**: periodo de trabajo donde se desarrolla una parte del producto
* **Product Backlog**: lista priorizada de funcionalidades
* **Sprint Backlog**: tareas seleccionadas para el sprint actual
* **Daily Scrum**: reunión diaria breve para revisar el progreso
* **Sprint Review**: revisión del trabajo realizado
* **Sprint Retrospective**: análisis de mejoras del proceso

Scrum permite tener un control constante del desarrollo y mejorar continuamente el trabajo del equipo.

## ¿Qué es Kanban?

Kanban es otra metodología Agile que se centra en la visualización del trabajo y en el flujo continuo de tareas.

Se basa en un tablero dividido en columnas que representan estados del trabajo, como por ejemplo:

* Backlog
* To Do
* In Progress
* Review
* Done

Cada tarea se representa como una tarjeta que se mueve entre columnas a medida que avanza.

### Características principales de Kanban

* Visualización clara del estado del proyecto
* Flujo continuo (no hay sprints)
* Limitación del trabajo en progreso (WIP)
* Flexibilidad para añadir o modificar tareas en cualquier momento

Kanban es especialmente útil para gestionar tareas de forma sencilla y mantener una visión global del trabajo.

---

## Diferencias entre Scrum y Kanban

| Aspecto       | Scrum                       | Kanban                           |
| ------------- | --------------------------- | -------------------------------- |
| Estructura    | Basado en sprints           | Flujo continuo                   |
| Planificación | Al inicio de cada sprint    | Continua                         |
| Cambios       | Limitados durante el sprint | Se permiten en cualquier momento |
| Roles         | Definidos                   | No obligatorios                  |
| Visualización | Opcional                    | Fundamental (tablero)            |

En resumen, Scrum es más estructurado y Kanban más flexible.

---

## ¿Cuándo usar cada metodología?

### Scrum

Es recomendable usar Scrum cuando:

* El proyecto es complejo
* Hay un equipo definido trabajando conjuntamente
* Se necesita planificación por ciclos
* Se busca mejora continua mediante retrospectivas

### Kanban

Es recomendable usar Kanban cuando:

* Las tareas son más pequeñas o variables
* Se necesita flexibilidad constante
* El trabajo no puede organizarse en sprints
* Se quiere una gestión visual sencilla

---

## Conclusión

Agile es una filosofía que permite desarrollar software de forma flexible y centrada en el usuario. Scrum y Kanban son dos formas de aplicar Agile, cada una con sus propias características.

Mientras Scrum aporta estructura y organización por ciclos, Kanban ofrece flexibilidad y visualización continua del trabajo. La elección entre uno u otro dependerá del tipo de proyecto y del equipo.

En este proyecto se ha optado por una organización más cercana a Kanban mediante el uso de un tablero de tareas, permitiendo una gestión flexible y visual del desarrollo.
